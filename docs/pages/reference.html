Service Limits & Quotas

Understanding the limits and quotas for CloudWatch Application Signals.

Trace Ingestion Limits

Per-Account Limits:

  • Trace segments per second: 100,000
  • Trace segments per burst: 200,000
  • Trace size: 64 KB per trace segment
  • Annotations per trace: 50
  • Metadata per trace: 50

Sampling Limits:

  • Reservoir size: 1 trace per second
  • Fixed rate: Up to 100% of traces
  • Custom rules: Up to 25 sampling rules per account

API Rate Limits

Application Signals APIs:

  • ListServices: 10 requests per second
  • GetService: 10 requests per second
  • ListOperations: 10 requests per second
  • GetOperation: 10 requests per second

X-Ray APIs:

  • GetTraceSummaries: 100 requests per second
  • BatchGetTraces: 200 requests per second
  • GetServiceGraph: 10 requests per second
  • PutTraceSegments: Based on trace ingestion limits

CloudWatch APIs:

  • GetMetricData: 50 requests per second
  • PutMetricData: 500 requests per second (per namespace)
  • ListMetrics: 25 requests per second

Data Retention

Trace Data:

  • Hot storage: 30 days (default)
  • Warm storage: 30-90 days (configurable)
  • Cold storage: 90 days to 1 year (configurable)

Metrics Data:

  • High resolution (≤ 60 seconds): 3 hours
  • 1-minute resolution: 15 days
  • 5-minute resolution: 63 days
  • 1-hour resolution: 15 months

RUM Data:

  • Event data: 30 days (default)
  • Extended retention: Up to 1 year (configurable)

Synthetics Data:

  • Run results: 90 days (default)
  • Canary artifacts: 7 days (default)

Query Limits

Trace Queries:

  • Time range: Up to 6 months
  • Results per query: 10,000 traces
  • Concurrent queries: 30 per account

Analytics Queries:

  • Time range: Up to 1 year
  • Data points: 500,000 per query
  • Concurrent queries: 20 per account

Service Quotas

Application Signals:

  • Services per account: 1,000
  • Operations per service: 100
  • Custom metrics: 30 per service

RUM:

  • App monitors per account: 20
  • Events per second: 10,000
  • Event size: 64 KB

Synthetics:

  • Canaries per account: 100
  • Canary execution time: 15 minutes maximum
  • Canary size: 1 GB (unzipped)

Glossary

Key terms and concepts in CloudWatch Application Signals.

Core Concepts

Application Performance Monitoring (APM)

The practice of monitoring and managing the performance and availability of software applications.

Distributed Tracing

A method of tracking requests as they flow through distributed systems, allowing you to understand the path and performance of requests across multiple services.

Service Map

A visual representation of your application's architecture showing how services connect and interact.

Span

A single operation within a trace, representing a unit of work such as a database query or API call.

Trace

A complete end-to-end journey of a request through your application, consisting of multiple spans.

Instrumentation Terms

Auto-Instrumentation

Automatic addition of observability code to applications without requiring code changes.

Manual Instrumentation

Explicit addition of tracing and metrics code to application source code.

OpenTelemetry

An open-source observability framework providing vendor-neutral APIs for instrumentation.

Sampling

The process of selecting which requests to trace to balance observability with cost and performance.

Context Propagation

The mechanism of passing trace context between services to maintain trace continuity.

Metrics and Monitoring

Golden Signals

The four key metrics for monitoring application health: latency, throughput, error rate, and saturation.

Latency

The time it takes for a request to be processed, typically measured in percentiles (P50, P90, P99).

Throughput

The rate at which requests are processed, usually measured as requests per second.

Error Rate

The percentage of requests that result in errors.

Saturation

The degree to which system resources are utilized.

Service Level Objective (SLO)

A target level of service quality agreed upon between service providers and consumers.

Service Level Indicator (SLI)

A measurable aspect of the service level provided, such as latency or availability.

RUM and Synthetics

Real User Monitoring (RUM)

Monitoring of actual user interactions with web applications in production.

Synthetic Monitoring

Automated testing of applications using simulated user interactions.

Canary

A synthetic monitoring script that runs automated tests against your application.

Core Web Vitals

Google's metrics for measuring user experience on the web: LCP, FID, and CLS.

Largest Contentful Paint (LCP)

The time it takes for the largest content element to be rendered on screen.

First Input Delay (FID)

The time from when a user first interacts with a page to when the browser responds.

Cumulative Layout Shift (CLS)

A measure of visual stability, quantifying unexpected layout shifts.

AWS-Specific Terms

Application Signals

AWS's unified observability solution combining APM, RUM, and synthetics.

X-Ray

AWS's distributed tracing service, now integrated with Application Signals.

CloudWatch

AWS's monitoring and observability service that collects metrics, logs, and traces.

ADOT (AWS Distro for OpenTelemetry)

AWS's distribution of OpenTelemetry optimized for AWS services.

Configuration Reference

Common configuration options and environment variables.

Environment Variables

OpenTelemetry Configuration:

  • OTEL_SERVICE_NAME - Name of your service
  • OTEL_TRACES_EXPORTER - Trace exporter (otlp, jaeger, zipkin)
  • OTEL_EXPORTER_OTLP_ENDPOINT - OTLP endpoint URL
  • OTEL_TRACES_SAMPLER - Sampling strategy
  • OTEL_TRACES_SAMPLER_ARG - Sampling parameter

AWS-Specific Configuration:

  • AWS_REGION - AWS region
  • AWS_XRAY_DAEMON_ADDRESS - X-Ray daemon address
  • AWS_APPLICATION_SIGNALS_ENABLED - Enable Application Signals

Sampling Configuration

Sampling Strategies:

  • always_on - Sample all traces
  • always_off - Sample no traces
  • traceidratio - Sample based on ratio (e.g., 0.1 for 10%)
  • parentbased_always_on - Respect parent sampling decision
  • parentbased_traceidratio - Parent-based ratio sampling

Cost Optimization

Strategies for controlling observability costs.

Sampling Strategies

  • Production: 5-10% sampling rate
  • Development: 50-100% sampling rate
  • Error traces: Always sample traces with errors
  • High-value operations: 100% sampling for critical paths

Data Retention Policies

  • Hot storage: Recent data for active troubleshooting
  • Warm storage: Historical data for trend analysis
  • Cold storage: Archive data for compliance
  • Data lifecycle: Automatic deletion based on retention policies

Resource Optimization

  • Batch exports: Reduce network overhead
  • Compression: Enable data compression
  • Filtering: Remove unnecessary spans and attributes
  • Buffering: Use appropriate buffer sizes
