Concepts & Fundamentals

Core Concepts

Understanding the fundamental concepts of Application Performance Monitoring (APM) is essential for effectively using CloudWatch Application Signals. This section covers the key building blocks that form the foundation of modern observability.

Distributed Tracing

Distributed tracing is the practice of tracking requests as they flow through distributed systems. In microservices architectures, a single user request might traverse multiple services, databases, and external APIs. Distributed tracing provides visibility into this entire journey.

Key Components:

  • Trace: A complete end-to-end journey of a request through your system
  • Span: A single operation within a trace (e.g., a database query or API call)
  • Trace Context: Metadata that connects spans across service boundaries

Traces, Spans, and Segments

These are the fundamental data structures used in distributed tracing:

Traces

A trace represents the complete path of a request through your application. It contains timing information, metadata, and all the spans that make up the request's journey.

Spans

A span represents a single unit of work within a trace. Each span has:

  • A name and operation type
  • Start and end timestamps
  • Tags and attributes for additional context
  • References to parent spans

Segments

In AWS X-Ray terminology, segments are the basic units of trace data. Each segment corresponds to a span and contains information about the work done for that part of the trace.

Service Maps

Service maps provide a visual representation of your application's architecture and dependencies. They show:

  • How services connect to each other
  • Traffic flow between services
  • Performance metrics for each service
  • Error rates and latency patterns

Service maps are automatically generated from trace data and help you understand your system's topology at a glance.

Instrumentation

Instrumentation is the process of adding observability code to your applications. There are several approaches:

Auto-Instrumentation

Automatic instrumentation requires no code changes. Libraries and agents automatically instrument common frameworks and libraries.

Manual Instrumentation

Manual instrumentation involves adding tracing code to your application. This provides fine-grained control but requires more development effort.

OpenTelemetry

OpenTelemetry is an open-source observability framework that provides vendor-neutral APIs for instrumentation. It supports multiple programming languages and can export data to various backends.

Data Model

Application Signals uses a structured data model to organize telemetry data:

Metrics

  • Latency: Response time measurements (P50, P90, P99)
  • Throughput: Request rate (requests per second)
  • Error Rate: Percentage of failed requests
  • Saturation: Resource utilization (CPU, memory, etc.)

Traces

Structured trace data with spans, tags, and timing information.

Logs

Application logs correlated with traces and metrics for complete context.

Sampling Strategies

Sampling controls which requests are traced to balance observability with cost and performance:

Head Sampling

Decides whether to trace a request at the beginning of its lifecycle.

Tail Sampling

Makes sampling decisions after collecting trace data, allowing for more intelligent sampling based on request characteristics.

Rate-Based Sampling

Traces a fixed percentage of requests (e.g., 10% of all requests).
