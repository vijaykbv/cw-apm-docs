What is CloudWatch APM?

CloudWatch Application Performance Monitoring (APM) is a comprehensive observability solution built on Application Signals that provides end-to-end visibility across your applications — from backend services to frontend user experiences.

Key capabilities include automatic service discovery, distributed tracing, real-time metrics, service maps, SLO tracking, and intelligent alerting.

The Three Pillars of Application Signals

Application Performance Monitoring (APM)

Monitor backend services, APIs, and microservices. Automatic instrumentation and distributed tracing reveal request flows and bottlenecks.

  • Distributed traces and service dependency maps
  • Performance metrics: latency (P50/P90/P99), throughput, error rates
  • Automatic service discovery for ECS, EKS, Lambda, and EC2

Real User Monitoring (RUM)

Capture real user experiences across browsers, devices, and geographies. Track page performance, JS errors, and Core Web Vitals.

Synthetic Monitoring

Run scheduled tests and scripted browser checks from multiple regions to detect issues before users do.

Getting Started Path

  1. Phase 1 — Start with APM (Week 1): Enable Application Signals for backend services and establish performance baselines.
  2. Phase 2 — Add Synthetics (Week 2–3): Create canaries for critical journeys and set up proactive monitoring.
  3. Phase 3 — Enable RUM (Week 4+): Add RUM to your web application and correlate frontend metrics with backend traces.

Developers: Instrument Your First Application

In ~15 minutes you'll have APM traces and metrics collecting for supported runtimes with zero code changes using auto-instrumentation.

Prerequisites

  • Application running on AWS (ECS, EKS, EC2, or Lambda)
  • Supported runtime: Java, Python, .NET, Node.js, or Go
  • IAM permissions to enable Application Signals

Step 1: Choose Instrumentation

  • Auto-Instrumentation — Zero code changes; recommended for quick starts.
  • OpenTelemetry Manual Instrumentation — Full control and portability.
  • AWS Distro for OpenTelemetry (ADOT) — AWS-optimized OTel collector.

Step 2: Enable Application Signals

Follow the platform-specific enablement instructions for EKS, ECS, or Lambda in the developer guide. After enabling, wait 5–10 minutes for data to appear in the console.

Step 3: Verify APM Data

  1. Open the CloudWatch Console → Application Signals
  2. Check the Service Map for discovered services
  3. Click a service to view distributed traces and metrics

Troubleshooting

  • APM traces missing: verify IAM permissions and that the agent/collector is running.
  • RUM not appearing: ensure the RUM snippet loads and app ID/region are correct.
  • Synthetics failing: check canary logs and network access from the canary VPC.

Operators: Monitor Production Systems

Set up monitoring, alerting, and dashboards across APM, RUM, and Synthetics.

Checklist

  • Service map includes all critical services
  • APM dashboards for latency, errors, throughput
  • SLOs and composite alerts spanning signals
  • Canaries for critical journeys and multi-region monitoring

Dashboards & Alerts

Create a unified dashboard using the Application Signals template and add widgets for APM golden signals, RUM metrics, and Synthetics health. Use composite alarms to correlate across signals.

Architects: Design Your Observability Strategy

Choose adoption, instrumentation, and retention strategies that scale with your organization.

Decision Framework

  • Adoption: Start with APM, add Synthetics, then RUM.
  • Instrumentation: OpenTelemetry for portability; AWS native for faster time-to-value.
  • Multi-Cloud: Consider OTel bridge if you need portability + AWS integration.
  • Retention: Use hot/warm/cold tiers and archive traces to S3 for cost control.

Patterns & Examples

Common patterns include EKS microservices with ADOT, serverless Lambda instrumentation via layers, and hybrid setups using OTel collectors.
